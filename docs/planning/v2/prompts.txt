## Task 2.1.1: Create Data Models

**File:** `backend/app/models/appointment.py`

Create Pydantic models for:

1. **AppointmentContext** — User's selections from Step 1
   - `appointment_type`: enum (new_provider, established_relationship)
   - `goal`: enum (understand_where_i_am, discuss_starting_hrt, evaluate_current_treatment, address_specific_symptom)
   - `dismissed_before`: enum (no, once_or_twice, multiple_times)

2. **AppointmentPrep** — Full appointment prep data
   - `user_id`: str (UUID)
   - `context`: AppointmentContext
   - `narrative`: str (LLM-generated summary of symptoms, editable by user)
   - `concerns`: list[str] (prioritized list of concerns from user)
   - `scenarios`: list[dict] (scenario cards with suggestions)
   - `created_at`: datetime
   - `updated_at`: datetime

3. **ProviderSummary** — One-page clinical overview
   - `content`: str (markdown/HTML, safe to share with provider)
   - `generated_at`: datetime

4. **PersonalCheatSheet** — Prioritized concerns + conversation anchors
   - `content`: str (markdown, more candid than provider summary)
   - `generated_at`: datetime

5. **AppointmentPrepOutput** — Combined outputs
   - `provider_summary`: ProviderSummary
   - `personal_cheat_sheet`: PersonalCheatSheet
   - `generated_at`: datetime

**Requirements:**
- Use Pydantic BaseModel or similar
- Include type hints on all fields
- Include docstrings explaining what each field represents
- Use enums for appointment_type, goal, dismissed_before (not strings)
- Include optional fields where appropriate (e.g., scenarios might not exist until Step 4)
- Follow the pattern from existing models (e.g., ChatMessage in app/models/chat.py)

## Task 2.1.2: Create Database Migration

**File:** `backend/app/migrations/add_appointment_prep_tables.sql`

Create SQL migration for:

1. **appointment_prep_contexts** table
   - `id`: uuid (primary key)
   - `user_id`: uuid (foreign key to auth.users, cascade delete)
   - `appointment_type`: text enum (new_provider, established_relationship)
   - `goal`: text enum (understand_where_i_am, discuss_starting_hrt, evaluate_current_treatment, address_specific_symptom)
   - `dismissed_before`: text enum (no, once_or_twice, multiple_times)
   - `created_at`: timestamp (default now)
   - `updated_at`: timestamp (default now)

2. **appointment_prep_outputs** table
   - `id`: uuid (primary key)
   - `user_id`: uuid (foreign key to auth.users, cascade delete)
   - `context_id`: uuid (foreign key to appointment_prep_contexts, cascade delete)
   - `provider_summary_content`: text (markdown/HTML)
   - `provider_summary_generated_at`: timestamp
   - `personal_cheat_sheet_content`: text (markdown)
   - `personal_cheat_sheet_generated_at`: timestamp
   - `created_at`: timestamp (default now)
   - `updated_at`: timestamp (default now)

**Security Requirements:**
- Add RLS policies so users can only see their own contexts and outputs
- Use authenticated role for all policies
- Policy: SELECT/INSERT/UPDATE/DELETE only where user_id = auth.uid()

**Quality Requirements:**
- Add indexes on (user_id, created_at) for common queries
- Add foreign key constraints with CASCADE DELETE
- Include comments on tables explaining purpose
- Ensure migration is idempotent (can run multiple times safely)
- Include timestamp columns (created_at, updated_at) on all tables

**Testing:**
- Migration should run without errors
- RLS policies should prevent users from accessing others' data
- Verify schema with: `\\d appointment_prep_contexts` and `\\d appointment_prep_outputs`
